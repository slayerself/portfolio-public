Pronostico eolico day-ahead con datos abiertos

Sistema operacional para una planta de 36 MW con emision diaria a las 08:00 hora Chile. El proyecto combina una correccion estadistica del pronostico de viento, un modelo de potencia por regimenes operativos (MOE) y una comparacion directa contra la programacion historica del Coordinador.

Que se construyo
Pronostico operativo de 24 horas

El flujo toma datos abiertos, corrige el viento pronosticado y estima la potencia inyectada para las 24 horas siguientes desde una emision fija diaria. Ademas del pronostico central, entrega cuantiles q25/q50/q75 para representar incertidumbre.

Como se evaluo
Comparacion justa con referencia externa

Se comparo contra la version diaria publicada por el Coordinador, usando la misma ventana operacional y las mismas metricas que importan en la practica: MAEn, RMSEn y BIASn.

Pregunta del proyecto

La pregunta central fue simple: ¿se puede construir un pronostico eolico operacional competitivo usando solo datos abiertos?

Desafio

Pronosticar para operar, no solo para entrenar

El problema no era obtener una metrica aislada en laboratorio, sino producir una estimacion diaria que pudiera compararse en igualdad de condiciones con una referencia real del sistema.

Restriccion

Datos abiertos y observabilidad limitada

El proyecto se resolvio sin SCADA, sin disponibilidad de turbinas y sin viento a altura de rotor, que son fuentes de informacion normalmente clave en un entorno industrial.

Criterio de exito

Comparacion justa con el Coordinador

La solucion debia sostenerse contra las metricas operacionales del Coordinador y bajo la misma logica de emision diaria, no contra una comparacion armada a posteriori.

Problema y restricciones

El objetivo no fue entrenar un modelo aislado, sino construir un sistema day-ahead comparable con una operacion real: misma hora de emision, misma ventana de validacion y metricas compatibles con el Coordinador.

Decisiones clave del proyecto

Estas definiciones marcaron el proyecto completo. Sin ellas, la comparacion final habria sido menos util y potencialmente enganosa.

Operacion

Emision fija a las 08:00

Todo el sistema se diseno para una emision diaria consistente en hora Chile, con una sola salida operativa por dia.

Horizonte

24 horas comparables

La ventana se fijo desde las 09:00 del mismo dia hasta las 08:00 del siguiente, para que la comparacion con el Coordinador fuera equivalente.

Rigor

Sin mezclar versiones futuras

Cada hora se evaluo con la version diaria que realmente estaba disponible al momento de emitir, evitando mejoras artificiales por usar informacion mas nueva.

Pipeline del sistema

El flujo se organizo en tres capas: correccion del viento, modelado de potencia y benchmark externo.

Correccion de viento

Calibrar primero el viento

OpenMeteo entregaba una base util, pero no lo suficientemente alineada con el sitio como para pasar directo a potencia. El bloque MOS se uso para corregir sesgo local y estabilizar la senal meteorologica antes del modelo final.

  • Corrige el viento forecast antes de mapearlo a potencia.
  • Reduce error sistematico por sitio, hora y horizonte.
  • Entrega una entrada mas consistente para el bloque de potencia.
Modelo por regimenes

No todas las horas operan igual

Un solo modelo era demasiado rigido para transiciones, recortes y eventos de caida brusca. El modelo por regimenes permitio separar familias operativas sin romper la salida unica que se necesita para uso diario.

  • Distingue comportamientos normales y regimenes especiales.
  • Usa un selector de regimen y submodelos especializados en una prediccion unica.
  • Sirvio para estudiar rampas y errores por familia, no solo error global.

La arquitectura se penso para un caso operacional real y no para una validacion academica desconectada de la emision diaria.

Metodologia y control anti-leak

La validez del proyecto dependia de respetar la logica de emision y de evitar mezclar informacion futura.

1

Split por momento de emision

La separacion temporal se definio por la version diaria publicada, no por la hora futura que se queria pronosticar.

2

Ventana operacional fija

El caso se fijo en 08:00 America/Santiago con horizonte horario de 24 horas desde la emision.

3

Calibracion solo en validacion

Los ajustes finales se calibraron en la etapa de validacion, nunca usando la ventana de prueba.

4

Benchmark por version diaria

La comparacion externa se hizo contra el archivo diario del Coordinador, no mezclando varias versiones para una misma hora.

Como leer las metricas

Las metricas se seleccionaron por su relevancia operacional. Cada una responde a una pregunta distinta sobre la calidad del pronostico.

MAEn

Error medio esperado

Resume cuanto se equivoca, en promedio, el pronostico respecto a la potencia instalada. Es la metrica mas directa para comparar desempeno global entre alternativas.

RMSEn

Peso de los errores grandes

Penaliza con mas fuerza las desviaciones grandes. Sirve para detectar si el modelo falla de manera especialmente severa en ciertas horas o eventos.

BIASn

Tendencia a sobre o subestimar

Indica si el modelo tiende a quedar sistematicamente por encima o por debajo de la potencia real. Es importante para distinguir error aleatorio de sesgo estructural.

Mejoras evaluadas

Las mejoras evaluadas respondieron a hipotesis tecnicas concretas sobre rampas, estabilidad o seleccion del submodelo de potencia.

A

Mejora enfocada en rampas fuertes

Se reforzo el submodelo de horas con recorte usando historial reciente disponible al momento de emitir.

B

Activacion selectiva del submodelo

Se uso la mejora solo cuando el sistema detectaba condiciones propicias, para evitar degradacion fuera de los casos objetivo.

C

Selector jerarquico con historial reciente

Se evaluo si la informacion temporal disponible al emitir ayudaba a elegir mejor el regimen operativo.

Hallazgos principales

Las variantes nuevas mejoraron estabilidad, sesgo y algunas metricas secundarias, pero no aparecio una mejora robusta en MAEn. Por eso la solucion base final no se reemplazo.

Hallazgo

El limite paso a ser de observabilidad

Sin SCADA, sin disponibilidad y sin viento a altura de rotor, el margen de mejora adicional en NMAE se vuelve mucho mas estrecho que el margen aparente de la arquitectura.

Figuras que guiaron decisiones

El proyecto no se apoyo solo en metricas globales. Estas dos vistas fueron claves para decidir que explorar y que no seguir persiguiendo.

El estudio de rampas mostro que los eventos mas fuertes se concentraban en horas con comportamiento de recorte. Eso justifico experimentar con submodelos especificos y no multiplicar alternativas sin una separacion real.
El historial reciente disponible al emitir entro entre las variables mas influyentes del selector jerarquico. Habia informacion util para mejorar la eleccion del submodelo, pero no la suficiente para cerrar por si sola la brecha de MAEn.

Uso practico del sistema

Aunque fue desarrollado como memoria y caso de estudio, el sistema ya quedo planteado en una forma que sirve para evaluacion operativa y para futuras extensiones.

Programacion

Apoyo a la estimacion diaria

Entrega una proyeccion de 24 horas con una hora fija de emision, lo que permite compararlo de forma ordenada con otras referencias operativas.

Benchmark

Linea base con datos abiertos

Permite medir cuanto se puede lograr sin telemetria interna y deja un punto de referencia claro para futuras mejoras con mejor informacion de planta.

Escalabilidad

Base para una version mas industrial

Si se agregan SCADA, disponibilidad o mejores variables meteorologicas, la estructura ya esta preparada para absorber esa informacion sin rehacer el proyecto desde cero.

Herramientas utilizadas

El proyecto se implemento con una base tecnica simple y reproducible, orientada a procesamiento de datos, modelado estadistico y automatizacion de corridas.

Lenguaje y entorno

Python y PowerShell

Python se uso para el procesamiento y modelado; PowerShell para runners, validaciones y ejecucion reproducible.

Datos y modelado

pandas, xarray, scikit-learn y LightGBM

Estas herramientas sostuvieron la preparacion de datos, la calibracion del viento y el modelado de potencia por regimenes.

Artefactos y visualizacion

parquet, JSON y matplotlib

Se usaron formatos simples para artefactos reproducibles y figuras que ayudaran a interpretar resultados y tomar decisiones.

Conclusion ejecutiva

El proyecto logro un sistema reproducible, comparable con operacion real y suficientemente riguroso como para medir su desempeno frente a una referencia externa consolidada.

Se logro

Un flujo operacional serio con datos abiertos

Se construyo un pronostico day-ahead completo, con correccion de viento, modelo de potencia, comparacion externa y artefactos reproducibles.

No se logro

Superar al Coordinador en la ventana comparable

La referencia externa mantuvo ventaja en las metricas principales, aun cuando el modelo propio fue competitivo y metodologicamente valido.

Siguiente salto

Mejor observabilidad, no solo mas arquitectura

Para cerrar la brecha restante haria falta mejor informacion operacional, como SCADA, disponibilidad de turbinas y viento mas representativo del rotor.
